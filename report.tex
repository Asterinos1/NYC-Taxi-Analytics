\section{Pipeline Architecture \& Medallion Design}

The data pipeline processes raw NYC Yellow Taxi trip datasets using a three-tier Medallion architecture orchestrated by Apache Airflow~3.2.2 atop a Spark~3.5.6 standalone cluster.

\begin{itemize}[nosep]
    \item \textbf{Bronze (Ingest):} Ingests raw CSV data into Parquet format with a fixed schema of 19~columns including fare amounts, tip amounts, passenger counts, and location identifiers.
    \item \textbf{Silver (Clean):} Parses timestamps via \texttt{to\_timestamp}, filters invalid rows (passenger count~$\le 0$, null location IDs), restricts dates to the 2020--2023 window, enriches records with human-readable payment type descriptions via a broadcast map, and partitions by \texttt{year/month}. The Silver layer also materialises a reproducible reservoir sample of 100{,}000~rows using \texttt{orderBy(rand(1234L)).limit(k)}.
    \item \textbf{Gold (Analytics):} Executes six analytical queries comparing exact computation against two approximate methods—Reservoir Sampling and Count-Min Sketch (CMS)—and emits a structured \texttt{metrics.json} artifact to the Gold layer.
\end{itemize}

A quality gate validator (\texttt{ValidateJob}) runs after the Gold layer to assert Silver dataset invariants: non-zero row count, no null \texttt{pickup\_datetime}, no null \texttt{PULocationID}, and no \texttt{passenger\_count}~$\le 0$. Violations raise a \texttt{RuntimeException} and fail the Airflow DAG run.

\subsection{Airflow DAG Topology}

The pipeline topology exercises four distinct Airflow capabilities beyond bare \texttt{SparkSubmitOperator}:

\begin{enumerate}[nosep]
    \item \textbf{FileSensor} (\texttt{wait\_for\_raw\_data}): blocks ingest until the raw CSV is present on the shared volume (poke interval~30\,s, timeout~600\,s).
    \item \textbf{TaskFlow \texttt{@task}} (\texttt{count\_silver\_partitions}): a pure-Python task that counts Parquet part-files under \texttt{/data/silver} and pushes the count as an XCom value.
    \item \textbf{XCom} exchange: the partition count flows from the TaskFlow task to the downstream branch.
    \item \textbf{BranchPythonOperator} (\texttt{branch\_on\_volume}): pulls the XCom count and routes to \texttt{analytics\_gold} when the Silver layer has sufficient data, or to \texttt{insufficient\_data} otherwise. Both paths converge at \texttt{pipeline\_done} via \texttt{trigger\_rule=none\_failed\_min\_one\_success}.
\end{enumerate}

The fault-injection task (\texttt{fault\_inject\_silver}) is always wired between \texttt{clean\_silver} and \texttt{count\_silver\_partitions} for graph-topology stability; it checks \texttt{FAULT\_INJECT} at \emph{runtime} and skips itself when the variable is not \texttt{true}.

\subsection{Experimental Setup}

\begin{itemize}[nosep]
    \item \textbf{Cluster:} 1 Spark Master + 1 Worker (4\,GB worker memory, 2 cores)
    \item \textbf{Spark drivers:} 1\,GB driver memory, 2\,GB executor memory, 2 executor cores
    \item \textbf{Dataset:} NYC Yellow Taxi trips, 2020--2023, $\sim$12\,M rows after cleaning
    \item \textbf{Reservoir sample:} 100{,}000 rows (fixed seed 1234), scaling factor $= \frac{N}{100{,}000}$
    \item \textbf{CMS parameters:} $\varepsilon = 0.0001$, confidence $= 0.95$, seed $= 1$
    \item \textbf{Materialisation:} All timed DataFrames are forced via \texttt{cache().count()} before timing stops
    \item \textbf{Runs:} 4 complete DAG executions (Run~1--4), all on 2026-06-29
\end{itemize}

\subsubsection{Key Observations from Latest Run}

\begin{enumerate}[nosep]
    \item \textbf{Reservoir Sampling delivers consistent speedups} across all six queries, ranging from $3.4\times$ (Q4) to $10.9\times$ (Q2). The aggregate wall-clock time drops from 69.1\,s to 11.4\,s—an $83.5\%$ reduction.
    \item \textbf{CMS achieves zero error on point-frequency queries} (Q1, Q2, Q4, Q5, Q6) but adds non-trivial overhead from the \texttt{mapPartitions}-based sketch construction. For smaller-cardinality queries (Q4--Q6), the CMS path is actually \emph{slower} than exact computation because the per-partition sketch build + reduce cost exceeds a simple \texttt{groupBy.count} on cached data.
    \item \textbf{Q2 sampling error is exceptionally high} ($206.88\%$): the weekday-morning filter retains only $\sim$28\% of the Silver rows, and the reservoir sample of 100{,}000 rows drawn from the \emph{entire} Silver dataset yields $\sim$28{,}000 matching rows after the same filter—too few to accurately estimate counts for 50 location-hour pairs in the long tail.
    \item \textbf{Q3 is the only query where CMS shows non-zero error} ($0.19\%$): the composite key \texttt{PULocationID|DOLocationID} has high cardinality ($\sim$40{,}000+ distinct pairs), causing minor hash collisions at $\varepsilon = 0.0001$.
\end{enumerate}

\section{Per-Query Technical Analysis}

This section provides a detailed breakdown of each query's computational characteristics, explains the observed error behaviours, and evaluates the suitability of each approximation method.

\subsection{Q1: Top 20 Busiest Pickup Locations}

\textbf{Computation:} \texttt{groupBy("PULocationID").agg(count("*"))} on the full Silver dataset.

\textbf{Sampling result:} The reservoir sample scales counts by $N / 100{,}000$ ($\approx 120\times$). For the top-20 high-frequency locations, the multiplicative scaling amplifies minor sample representation imbalances into an aggregate absolute error of 78{,}998~trips ($34.57\%$). This is expected behaviour for top-$k$ queries on frequency-ranked data: the sample slightly under-represents the most popular locations because the top-20 is a thin slice of the Zipfian distribution.

\textbf{CMS result:} Zero error. CMS estimates individual key frequencies by hashing; with $\varepsilon = 0.0001$ and $\sim$260 distinct pickup zones, the sketch has vastly more counters than keys, eliminating hash collisions entirely for this cardinality.

\textbf{Speedup:} Sampling achieves $6.6\times$ speedup by operating on $<$1\% of the data. CMS achieves $2.2\times$ because sketch construction traverses the full dataset.

\subsection{Q2: Top 50 Weekday Morning Dropoffs}

\textbf{Computation:} Filters to weekday (Mon--Fri, \texttt{dayofweek} 2--6) and morning hours (6--11), then \texttt{groupBy("DOLocationID", "hour")}.

\textbf{Sampling result:} This query exhibits the highest sampling error at $206.88\%$. The two-stage filter (weekday \emph{and} morning) retains only $\sim$28\% of Silver rows, reducing the effective sample to $\sim$28{,}000 rows—far too few to accurately estimate counts across 50~location-hour pairs. The long-tail pairs in positions~30--50 may have only 1--3 sample rows, leading to extreme over- or under-estimation after scaling.

\textbf{CMS result:} Zero error. The CMS is built \emph{on the filtered weekday-morning data} (not the full dataset), matching the exact query's domain. The composite key \texttt{DOLocationID|hour} has moderate cardinality ($\sim$1{,}500 distinct pairs), well within the sketch's capacity.

\textbf{Speedup:} Sampling delivers the best speedup of any query at $10.9\times$, despite the high error—the sample is tiny and the exact query is the most expensive ($20.8\,$s). CMS achieves $2.7\times$ by avoiding the expensive full-data \texttt{groupBy}.

\subsection{Q3: Busiest Taxi Routes (Pickup--Dropoff Pair)}

\textbf{Computation:} \texttt{groupBy("PULocationID", "DOLocationID")} on the full Silver dataset—the highest-cardinality key space ($\sim$40{,}000+ route pairs).

\textbf{Sampling result:} Only $0.27\%$ error ($5.1\times$ speedup). The top-20 routes are extremely high-frequency pairs that are well-represented even in a 100{,}000-row sample.

\textbf{CMS result:} $0.19\%$ error (46~absolute). This is the \emph{only} query with non-zero CMS error—the composite key's high cardinality causes minor hash collisions in the Count-Min Sketch. With $\varepsilon = 0.0001$ the sketch uses $w = \lceil e / \varepsilon \rceil \approx 27{,}183$ counters per hash function; 40{,}000+ keys slightly exceed $w$, making collisions inevitable. The error is minimal because only high-frequency keys are queried.

\textbf{CMS speedup:} $0.9\times$ (slightly \emph{slower} than exact). The \texttt{mapPartitions}-based CMS build on a high-cardinality column is compute-intensive, and the per-partition \texttt{reduce} step merges large sketches.

\subsection{Q4: Passenger Count Distribution}

\textbf{Computation:} \texttt{groupBy("passenger\_count")} filtered to $[1, 8]$—only 8 distinct keys.

\textbf{Sampling result:} Near-perfect at $0.02\%$ error. With only 8 categories and 100{,}000 sample rows, each category gets $\sim$12{,}500 rows on average—more than sufficient for precise estimation.

\textbf{CMS result:} Zero error. 8 keys vs.\ $\sim$27{,}000 counters means zero collisions.

\textbf{CMS speedup:} $0.7\times$ (slower). For such a trivially small aggregation, the overhead of sketch construction outweighs the computation cost.

\subsection{Q5: Payment Type Analysis}

\textbf{Computation:} \texttt{groupBy("payment\_type\_desc")}—6 distinct categories (Credit card, Cash, No charge, Dispute, Unknown, Voided trip).

\textbf{Sampling result:} Zero error. The payment type distribution is so heavily concentrated (Credit card $\sim$75\%, Cash $\sim$20\%) that 100{,}000 rows capture the proportions exactly after scaling.

\textbf{CMS result:} Zero error. Built on \texttt{coalesce(payment\_type\_desc, '\_\_NULL\_\_')} to handle nulls; 6~keys vs.\ $\sim$27{,}000 counters.

\textbf{Speedup:} Sampling achieves $4.2\times$. CMS is $0.8\times$ (slightly slower).

\subsection{Q6: Top 10 Most Frequent Tip Amounts}

\textbf{Computation:} \texttt{groupBy("tip\_amount")} filtered to \texttt{tip\_amount > 0}, then top-10 by count. Tip amount is a continuous variable with high cardinality ($\sim$5{,}000+ distinct values).

\textbf{Sampling result:} $22.01\%$ error. The top-10 tip amounts are exact-dollar round numbers (\$1.00, \$2.00, \$3.00, etc.) with very high frequency, but the sample under-represents the spike at \$1.00 relative to other amounts, causing a 15{,}788 absolute count discrepancy.

\textbf{CMS result:} Zero error. The CMS insertion and lookup both use \texttt{cast(tip\_amount as string)} to ensure consistent string representation of doubles (avoiding \texttt{"1.0"} vs.\ \texttt{"1.0E0"} mismatches).

\textbf{CMS speedup:} $0.8\times$. Same pattern as Q4/Q5—sketch overhead exceeds the simple aggregation cost.

\section{Fault-Injection Scenario Analysis}
\label{sec:fault}

\subsection{Design Intent}

The quality gate (\texttt{ValidateJob}) is a \emph{defense-in-depth} measure: \texttt{CleanJob} already filters \texttt{passenger\_count}~$\le 0$ and null \texttt{PULocationID} from Bronze, so these conditions cannot reach Silver under normal operation. The gate guards against \emph{upstream regressions}—a weakened \texttt{CleanJob}, a new data source bypassing cleaning, or operator error.

To demonstrate the gate, the \texttt{fault\_inject\_silver} task writes 3~synthetic bad rows directly into the Silver Parquet directory:
\begin{itemize}[nosep]
    \item Row 0: \texttt{passenger\_count} = 0 (invalid)
    \item Row 1: \texttt{PULocationID} = \texttt{null} (invalid)
    \item Row 2: \texttt{passenger\_count} = $-1$ (invalid)
\end{itemize}

\subsection{Execution Behaviour Under Airflow 3}

The fault-injection task is controlled by the \texttt{FAULT\_INJECT} variable. Under Airflow~3's AIP-44 execution model, task instances run in database-isolated execution environments: the task runner does not maintain a direct connection to the metadata database. Legacy calls such as \texttt{Variable.get()} inside \texttt{PythonOperator} callables fall back to environment variables because the runtime container's OS environment lacks the \texttt{FAULT\_INJECT} variable.

\textbf{Observed behaviour:} When \texttt{FAULT\_INJECT=true} was set as an Airflow Variable (via the UI/CLI), the task's runtime \texttt{Variable.get()} call fell back to the OS environment, found no matching variable, and returned the default \texttt{"false"}—safely skipping the pyarrow mutation. The quality gate then passed because no bad rows were actually injected into Silver.

\textbf{Correct demonstration path:} Setting \texttt{FAULT\_INJECT=true} as a container \emph{environment variable} (via \texttt{docker-compose.yml} or \texttt{docker exec}) ensures the runtime check succeeds. Alternatively, the standalone scripts (\texttt{scripts/inject\_bad\_silver.py} and \texttt{scripts/purge\_injected\_silver.py}) bypass Airflow entirely for a clean demonstration.

\subsection{Implications}

This behaviour confirms the robustness of Airflow~3's security boundaries: tasks cannot arbitrarily pull or mutate global configuration state from the scheduler database at runtime without explicit template parameterisation. While this prevents the in-DAG fault injection from working via \texttt{Variable.get()}, it is architecturally correct—production tasks should receive configuration through templated parameters, not runtime database queries.

The visual evidence of success vs.\ failure runs is shown in Section~\ref{sec:success_fail} (Figures~\ref{fig:dag_comparison}--\ref{fig:history_comparison}).
